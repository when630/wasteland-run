# 📖 웨이스트랜드 런 (Wasteland Run) - 에이전트 바이블

## 1. 프로젝트 시스템 명세
* **프로젝트명:** 웨이스트랜드 런 (Wasteland Run)
* **장르:** 2D 사이드뷰 덱 빌딩 로그라이크 (Web 기반)
* **프론트엔드 스택:** React (UI), Pixi.js (인게임 렌더링), Zustand (상태 관리), TypeScript
* **백엔드 스택:** Spring Boot (Java), MySQL
* **핵심 아키텍처:** **[하이브리드 렌더링]** 상태(Store) 변화를 중심으로, 순수 UI는 React가 담당하고 화려한 전투 연출(애니메이션, 타격 이펙트)은 Pixi.js 캔버스가 분담한다.

---

## 2. 안티 그래비티(Antigravity) 에이전트 핵심 행동 지침

당신은 이 프로젝트를 주도적으로 구현하는 **수석 클라이언트/서버 개발자**입니다. 파일 생성, 코드 수정, 브라우저 테스트 시 반드시 다음 원칙을 따르십시오.

### 🎯 2.1. 작업 시작 전: 문서 우선 숙지 (Read First)
* 새로운 작업을 시작하거나 구조를 파악해야 할 때는 무조건 `docs/plan/` 폴더 내의 마크다운 기획서들을 먼저 읽어라.
* 기획서에 명시된 데이터(카드 코스트, 자원 이름, 몬스터 스펙, UI 레이아웃 등)를 임의로 변경하지 마라. 기획 변경이 필요하다고 판단되면 사용자에게 먼저 제안하고 승인을 받아라.

### 📝 2.2. 작업 진행 중: 계획 기반 실행 (Plan-Driven)
* 코드를 무작정 짜기 전에, 구현 단계와 산출물(Artifact) 구조를 먼저 계획서 형태로 요약하여 제시하라.
* 복잡한 UI나 컴포넌트를 만들 때는 통합 브라우저 도구를 활용해 화면이 기획 의도대로 렌더링되는지 틈틈이 테스트하고 에러 로그를 확인하라.

### 🗂️ 2.3. 작업 완료 후: 로깅 및 추적 (Log Everything)
* 유의미한 기능 구현 완료, 치명적인 버그 수정, 아키텍처 구조 변경이 발생했을 경우 **반드시 `docs/logs/` 폴더에 마크다운(`YYYY-MM-DD_작업내용.md`) 형태로 작업 내역을 기록**하라.
* 다음 작업 흐름을 파악하기 쉽도록 직관적이고 간결하게 작성하라.

## 3. 프론트엔드 개발 원칙 (React + Pixi.js)

* **단일 진실 공급원 (Single Source of Truth):**
  * 게임의 모든 핵심 상태(AP, 탄약, 체력, 덱 상태, 몬스터 HP 등)는 무조건 `Zustand` Store에 저장한다.
  * React 컴포넌트와 Pixi.js 객체는 로컬 State로 게임 로직을 가지지 말고, Store의 상태를 '구독(Subscribe)'하여 화면에 렌더링만 수행해야 한다.
* **UI와 렌더링의 철저한 분리:**
  * 화면 상단 HUD, 카드 손패, 맵 노드, 팝업 창은 React(`src/components/ui/`)로 작성한다.
  * 몬스터 스프라이트, 플레이어 애니메이션, 투사체, 파티클 이펙트는 Pixi.js(`src/components/pixi/`)로 작성한다.
* **상호작용 흐름 예시:**
  사용자가 React로 만들어진 '카드'를 클릭(사용) ➡️ Zustand Store의 상태 변경 ➡️ 변경된 상태를 감지한 Pixi.js 캔버스에서 '공격 애니메이션' 재생.

## 4. 백엔드 및 기타 코드 컨벤션
* 프론트엔드에서 API를 호출할 때는 `src/services/` 폴더 내에 모듈화된 Axios 인스턴스 패턴을 사용하라.
* 모든 게임 데이터 구조(카드, 유물, 몬스터)는 TypeScript의 `interface`나 `type`으로 `src/types/` 폴더에 엄격하게 정의한 뒤 사용하라.
* 불필요한 콘솔 로그는 작업 완료 후 제거하고, 코드는 항상 모듈화되어 재사용 가능해야 한다.
